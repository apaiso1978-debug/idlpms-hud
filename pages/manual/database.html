03 Scaling Architecture (10M Users)

Google Sheets Sharding (AppScript)

ใช้กลยุทธ์การแบ่งคูหาข้อมูล (Sharding) โดยใช้ Google Sheets และประมวลผลผ่าน AppScript เพื่อกระจายภาระงาน รองรับผู้ใช้งานหลัก 10,000,000 คน ได้อย่างมีประสิทธิภาพ

FIFO Google Drive Storage

ระบบจัดสรรพื้นที่เก็บไฟล์ด้วย Google Drive โดยใช้หลักการ FIFO (First-In, First-Out) เพื่อการจัดการพื้นที่อย่างคุ้มค่า และลดการสะสมไฟล์ที่ไม่ได้ใช้งานนาน

FIFO Memory Guard (Client)

จำกัดจำนวน DOM elements ที่แสดงผล (Limit 50) เพื่อป้องกันความหน่วงของบราวเซอร์ และการใช้หน่วยความจำเกินความจำเป็น

Multi-District Balancing

การจัดการข้อมูลแยกรายเขตพื้นที่ฯ (ESA District Partitioning) เพื่อประสิทธิภาพสูงสุดในการเรียกใช้และจัดเก็บข้อมูลตามพื้นที่

04 โครงสร้างข้อมูลบทเรียน (Pedagogical Data Schema)

pedagogicalData (Standard Object)

โครงสร้างหลักที่ได้จากการ Harvest ต้องรองรับความหลากหลายของข้อมูล DLTV ดังนี้:

// Objectives (Polymorphic)
objectives: { K: "", P: "", A: "" } || Array[]

// Core Content
indicator: "ส 4.x ...",
summary: "สรุปเนื้อหาสำคัญ...",
hlsUrl: "https://...",
// Assessment (🤖 AUTO)
quiz: [
  { question: "", options: [], answer: 0 }
],
masterChallenge: {
  scenario: "ถ้านักเรียน...",
  criteria: []
}

Harvesting Rule: Sidebar Verification

ต้องเขียนบันทึก Subject Name, Year, และ Semester จาก Sidebar ขวา ลงใน Metadata ของบทเรียนเสมอ เพื่อใช้เป็น Primary Key ในการดึง HLS Stream ที่ถูกต้อง
